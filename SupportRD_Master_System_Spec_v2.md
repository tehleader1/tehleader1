# SupportRD Master System Spec v2

## Core Identity

SupportRD is one connected Remote-first platform.

- The `Remote` is the master shell.
- Every feature should open from the Remote, stay emotionally connected to the Remote, and return cleanly back to the Remote.
- No action should feel dead, loose, or disconnected from account identity.

## Remote Rules

- The Remote opens in on-the-go ready mode.
- The Remote is sealed as the main gameplay shell for the site.
- Every button must lead to a real result.
- No accidental multiple-click opening loops.
- Every route must preserve SupportRD identity and return path.

## Account System

- One account owns:
  - profile
  - diary feed
  - studio sessions
  - purchase memory
  - premium unlocks
  - assistant memory
- Logged-out viewers can still watch public diary sessions.
- Guest access cannot break account ownership.

## Diary System

- Each live diary feed should carry:
  - `session_id`
  - `owner_email`
  - `live_slug`
  - `^^ profile_tag`
- Entry points:
  - direct URL
  - Diary Live Lobby
  - search by Name
  - search by Email
  - search by Tag
  - search by URL
- Public session view includes:
  - live screen
  - live comments
  - guest support/payment
  - big `X` back to Remote

## Studio System

- Studio belongs to the same account system.
- Premium Jake features remain locked until paid.
- Studio sessions should persist under the same SupportRD identity.
- Audio and video creation should feel like one booth.
- Studio must remain launchable from the Remote without losing route context.

## Aria / Jake System

- One shared intelligence backbone across:
  - moving icons
  - diary mode
  - hands free mode
  - studio support
- Assistant movement is route-aware.
- Assistant movement is scanner-aware.
- Assistant greeting should feel immediate and human.
- Assistant state should remain visible in the scanner/HUD.

## Settings Requirements

Settings cannot fall short. It is the practical pit stop for keeping the app usable on the trail.

- Login and account controls must be easy to reach.
- Push notification details must be configurable quickly.
- URL and social configuration must be editable without leaving the shell.
- Password / verification controls must stay serious and clear.
- Settings should feel relaxed, not stressful.
- A lightweight SupportRD comfort layer should live here:
  - short TV reel
  - fast route to FAQs
  - route back to Diary / Profile / Payments

### Settings Detailed Modules

- `Main Structure`
  - identity / login state
  - email verification state
  - password change flow
  - profile URL / live URL settings
- `Statistics`
  - last movement activity
  - latest session activity
  - push notification health
  - active route / panel health
- `General Options`
  - push notifications
  - social/contact links
  - comfort reel
  - quick-access controls
- `Contacts / Channels`
  - URL settings
  - feed channel details
  - support email / phone slots
  - diary lobby visibility state

## Profile Requirements

Profile cannot fall short. It is the serious identity lane.

- It must show:
  - identity confidence
  - general status reading
  - hair analysis
  - profile image
  - export history
- API access should be represented in Profile as:
  - detailed access scanner
  - safe system access visibility
  - no exposure of raw secret keys
- Profile scanner functions:
  - confirm identity
  - read general status from account memory
  - confirm hair analysis
  - store analysis into intelligent history
  - export analysis to PDF / DOCX

### Profile Detailed API Areas

- `Professional Identity`
  - identity confirmation based on account, route context, and safe IP-derived trust signals
  - wording should feel presentation-ready and meeting-ready
- `General Status Reading`
  - wellness / account stability summary from SupportRD history
  - should read like a calm last-minute system check
- `Hair Analysis`
  - image-backed hair reading
  - intelligent result history
  - export to `.pdf` and `.docx`
- `API Access Deck`
  - OpenAI armed state
  - Diary API armed state
  - Studio API armed state
  - Account API armed state
  - Shopify / revenue armed state
  - cloud / persistence armed state
  - secret values must never be exposed directly on the page

## Map Change Requirements

Map Change is a pit stop route, not just a decoration lane.

- It must include:
  - general information
  - actual map change controls
  - ad lane behavior
  - perks details on the side
- It should feel like:
  - route reset
  - visual recharge
  - guidance lane
- It should never feel disconnected from the Remote shell.

### Map Change Side Layout

- `General Information`
  - current map / route explanation
  - on-trail helper copy
- `Ads`
  - ad lane / promotions
  - premium support placements
- `Actual Map Change`
  - live theme / map rotation control
  - directional change buttons
- `Perks Details`
  - side card with member perks
  - shortcut to revenue lanes without breaking the flow

## FAQ Lounge Requirements

FAQ Lounge is the SupportRD view into the social world.

- It is not just a static FAQ page.
- It should feel like a relaxed support lounge while on the trail.
- It must include:
  - direct FAQs
  - quick support access
  - social-world feeling
  - 10-second TV reel comfort layer
  - route back to settings/profile/help actions
- It should help a person:
  - configure a URL
  - change push notification details
  - stay relaxed while they do it

### FAQ Lounge Personality

- it should feel like a relaxed support lounge while still being productive
- it should keep a `SupportRD on the trail` identity
- it should combine:
  - social-world feel
  - support answers
  - configuration comfort
  - short TV-style reel
  - direct route back to Settings / Profile / Diary / Payments

## Revenue Rules

- Revenue lanes must feel native to SupportRD.
- Revenue should not eject the user emotionally from the product.
- Revenue must include:
  - premium upgrades
  - diary support
  - studio premium
  - custom orders
  - session-linked buying
- Checkout return must tie purchases back to account ownership.

## Scanner Rules

- The scanner must track:
  - route
  - panel
  - visible controls
  - active anchors
  - movement zones
  - button health
  - API readiness
  - account readiness
  - diary/studio/profile/settings relevance
- Purpose:
  - prove what is live
  - expose what is dead
  - keep Aria/Jake movement intelligent

### Scanner Detail Requirements

- scanner should report detailed system information under access:
  - account system state
  - Studio system state
  - Diary system state
  - voice system state
  - revenue / Shopify state
- scanner should be the trusted view for:
  - why a feature is not opening
  - whether a button is armed
  - where Aria/Jake can move
  - what route boundaries are active

## SEO Rules

- Every route is a valid entrance into the SupportRD world.
- Main examples:
  - `/remote`

  - `/remote/studio`
  - `/remote/profile`
  - `/remote/settings`
  - `/remote/payments`

## Final Product Goal

SupportRD should feel like:

- one world
- one identity
- one shell
- one voice
- many connected zones

If one boundary breaks, the world feels fake.

If all boundaries connect, SupportRD feels like one real product.
